# Spec: Smart URL Field — Admin UI Redesign

## Complexity
**medium**

Reasoning: 3-5 requirements, modifies existing field across 6 integration points, requires first-ever custom Payload admin UI component, changes data model from multi-field to single string. No new flows or multi-step journeys — it's a field replacement within existing forms.

## Goal
Replace the current multi-field smart-link (select + text + select with hardcoded options) with a single combobox/autocomplete field that stores a plain string and dynamically loads all available pages and sections — making link management intuitive and requiring minimal actions from admins.

## Background
- **Current state**: `smartLinkFields()` generates 3 Payload fields (`linkType`, `url`, `section`) with conditional visibility. Sections are hardcoded (14 options). Data is stored across multiple DB columns and resolved at render time by `resolveHref()`. Used in 6 blocks/globals.
- **Problem**: UX is confusing — admins must switch between "custom" and "section" modes, sections are hardcoded and don't reflect actual page content, multiple fields for a single concept.
- **Target state**: A single text field storing a plain string (`https://...` for external, `/slug` or `/slug#section` for internal). Admin UI shows a combobox: type freely for external URLs, or click/search to pick from dynamically-loaded pages and sections. No special prefix — internal links are just paths, ready to use as `href` directly.

## Requirements

### MUST (blockers — feature doesn't work without these)
- [ ] REQ-1: Single Payload field definition (`text` type with custom admin component) replacing the current 3-field pattern
- [ ] REQ-2: Custom admin React component (combobox/autocomplete) that works as both a free-text input and a dropdown picker
- [ ] REQ-3: Dynamic loading of all pages (title + slug) and their sections (block label + blockType) from Payload API on field focus/open
- [ ] REQ-4: Store value as a single plain string: `https://...` (external), `/slug` (internal page), `/slug#section-id` (internal section). No `internal:` prefix — values are valid `href` attributes directly. **The `#fragment` always uses `toKebabCase(blockType)`** — matching the current DOM `id` generated by `render-blocks.tsx`. If a page has two blocks of the same type, they produce the same fragment — this is a known limitation addressed in Out of Scope.
- [ ] REQ-5: Visual distinction in the field between an internal link (starts with `/`, displayed as chip/badge showing page/section name) and an external URL (starts with `http`, displayed as plain text)
- [ ] REQ-6: Update all integration points across 5 files / 8 call sites (hero ×1, process ×1, cta ×1, footer ×2, header ×3) to use the new field
- [ ] REQ-7: Simplify `resolveHref()` — the stored value is already a valid href (`/about#hero` or `https://...`), so it becomes a direct passthrough. Remove the old `linkType`/`url`/`section` parsing logic.
- [ ] REQ-8: Frontend rendering: external links (starts with `http`) use `<a>` with `target="_blank"` and `rel="noopener noreferrer"`. Internal links (starts with `/`) use Next.js `<Link>` component for client-side navigation.
- [ ] REQ-9: Field changes update only local Payload form state (not DB) — saving happens via the global Save button

### SHOULD (important but not blocking)
- [ ] REQ-10: Dropdown groups pages as top-level items with their sections as nested items (e.g. "About → Hero, About → Process")
- [ ] REQ-11: Search/filter within the dropdown by page name or section name
- [ ] REQ-12: When clicking on a field with an internal link, auto-select all text so the admin can immediately paste/type a replacement
- [ ] REQ-13: Show section labels (user-given `blockName`) in dropdown to distinguish multiple blocks of the same type on one page. Fall back to `blockType` when `blockName` is empty.

## UI States

| State | Description | Trigger |
|-------|-------------|---------|
| Empty | Text input with placeholder, dropdown available on focus | Field has no value |
| External URL | Plain text showing the URL | Value starts with `http://` or `https://` |
| Internal Link | Chip/badge showing page name (+ section name if applicable) | Value starts with `/` |
| Dropdown Open | Grouped list of pages and sections, with search | Field focused or chevron clicked |
| Loading | Spinner or skeleton in dropdown | Fetching pages/sections from API |
| Error | Inline error message | API fetch fails |

## Acceptance Criteria

### Scenario 1: Field initialization on page load
- **Given**: Admin opens a page/global that contains the smart URL field, and the field has a saved value `/about#hero`
- **When**: The page finishes loading
- **Then**: The field displays a chip/badge showing the page name "About" and section name "Hero" (resolved from slug and blockType). Pages/sections data loads when the dropdown is opened (not on page load).
- **DB Seed**: Create a page with `slug: "about"`, `title: "About"`, containing a `hero` block. Create another page (e.g. home) that uses a block with a smart link field set to `/about#hero`.

### Scenario 2: Adding an external URL to an empty field
- **Given**: The smart URL field is empty
- **When**: Admin clicks the field and pastes/types `https://www.google.com`
- **Then**: The field displays the URL as plain text, the page's Save button becomes active (dirty state). No confirmation dialogs or mode switches needed.
- **DB Seed**: Create a page with a block containing an empty smart link field.

### Scenario 3: Changing external URL to internal link
- **Given**: The field contains `https://www.google.com`
- **When**: Admin clicks the field and selects "About → Hero" from the dropdown
- **Then**: The field value changes to `/about#hero`, displayed as a chip/badge. Save button becomes active.
- **DB Seed**: Create a page with `slug: "about"`, `title: "About"`, containing a `hero` block. Create a second page with a block whose smart link = `https://www.google.com`.

### Scenario 4: Changing internal link to external URL
- **Given**: The field contains `/about#hero` (displayed as chip/badge)
- **When**: Admin clicks on the field — all text is selected — and pastes `https://example.com`
- **Then**: The chip is replaced with the new URL as plain text. Save button becomes active.
- **DB Seed**: Same as Scenario 3, but with the smart link field set to `/about#hero`.

### Scenario 5: Saving with page status preserved
- **Given**: A published page with a smart link field set to `https://old-url.com`
- **When**: Admin changes the field to `/about` and clicks Save
- **Then**: The page is saved to DB with the new link value. Page status remains "published" (no change to draft/published state).
- **DB Seed**: Create a published page with a block containing smart link = `https://old-url.com`. Create a page with `slug: "about"`.

### Scenario 6: Section names in dropdown reflect block labels
- **Given**: A page "Home" has two `process` blocks — one labeled "Our Process" and another labeled "How It Works"
- **When**: Admin opens the dropdown in a smart link field
- **Then**: Under "Home", both sections appear with their user-given labels: "Our Process (process)" and "How It Works (process)" — not just "process" twice.
- **DB Seed**: Use the existing "home" page (already contains multiple blocks). If needed for this scenario, ensure at least two blocks of the same type have distinct `blockName` values.

## Out of Scope
- Data migration from old multi-field format to new single-string format (will be a separate task)
- Backwards compatibility layer for old `{linkType, url, section}` data shape
- Validation of external URLs (e.g. checking if they return 200)
- Anchor/section ID deduplication when multiple blocks of the same type exist on a page
- Localization of dropdown labels (uses current locale page titles, but no special i18n work)
- Custom field validation beyond required/non-empty

## Design Decisions
- **`#fragment` identity**: Uses `toKebabCase(blockType)` to match existing DOM anchors. Duplicate block types on the same page produce the same fragment — this is accepted as a known limitation (fixing anchor uniqueness is out of scope).
- **No `internal:` prefix**: Stored values are valid hrefs directly (`/about#hero`, `https://...`). `resolveHref()` becomes a simple passthrough — no parsing needed.
- **Dropdown loading**: Fetch pages/sections on first dropdown open, cache for the admin session. Fresh data appears after page reload.
- **Block labels**: Payload CMS provides `blockName` on all blocks by default. Dropdown uses `blockName` when available, falls back to `blockType`.
